Persona card & lead-mode anatomy

Naming, behavior, and decision points for every block inside the red rectangle in the lead-mode sidebar — so we can talk about each piece independently and lock in when it should appear, animate, and collapse.

1 · Anatomy of the red rectangle

What looks like one "animated persona card" is actually six independently controlled blocks stacked inside the lead-mode sidebar. Each can appear, animate, or hide based on different triggers. Naming them clearly is the first step to controlling them.

  1. Persona card #persona · CSS class .persona The avatar pill identifying who's about to act. Persistent across all states (always visible, even when lead-mode is collapsed). Side determines color: advisor = green/tenant, client = Docusign purple, system = neutral, handoff = split gradient.
  2. Step label #step-label · CSS class .step-label The blue "SCENE N OF M · …" breadcrumb. Always present.
  3. Editorial headline #headline · CSS class .headline The Fraunces-italic title. Swells to 42px in lead-mode; shrinks to 22px Inter when lead-mode collapses.
  4. Lede #lede · CSS class .lede The single-paragraph elaboration under the headline.
  5. Lead hint .lead-hint (pseudo-element with pulsing dot) "Click anywhere to begin." Hidden by default; appears only after lead-mode has been showing for LEAD_HINT_DELAY_MS (1.2s) and the user hasn't moved.
  6. Persona-handoff cue #persona-handoff · CSS class .persona-handoff The cream-pink "Click or → to hand off to the next actor" box. Hidden by default; appears only when autoplay pauses at a persona side switch.

The whole red-bordered region as a unit is what the code calls lead-mode (CSS class .shell.lead-mode). The bounce-in animation when lead-mode enters is a separate class .is-popping on the sidebar element.

2 · State machine — when does each block appear?

Five states determine what's visible. Each block subscribes to a different combination.

Block Visible when… Animates when… Duration
1. Persona card always (every state) persona identity changes (side, name, or role) — slide-in bounce 0.5s
2. Step label always scene change — fades via sidebar fade 0.18s
3. Editorial headline always (size differs by state) scene/beat change — text swap; size lerps with lead-mode toggle 0.28s
4. Lede always scene/beat change — text swap 0.22s
5. Lead hint .shell.lead-mode-hinted is on added 1.2s after lead-mode enters, removed on stage interaction 0.55s fade
6. Persona-handoff cue #persona-handoff not [hidden] autoplay paused at a persona switch; cleared by click / → / Next / space 0.32s in
(panel itself) .shell.lead-mode is on scene change OR beat change (Next / Prev / arrow / autoplay tick); auto-collapses after a beat-length read window 3.0s – 6.5s

3 · The four lifecycle events

There are exactly four moments at which any of these blocks animate. If something is animating outside one of these four, that's a bug.

Event What fires What stays put
A. Scene change (tab click, end-of-scene autoplay) Lead-mode enters · sidebar pops · headline swells · iframe fades · lead hint armed (1.2s) Persona card only bounces if persona identity actually changed across the boundary
B. Beat change within a scene (Next/Prev/→/← /autoplay tick) Lead-mode re-enters · sidebar re-pops · headline + lede text swap · iframe stays mounted Persona card only bounces if persona changed; iframe doesn't reload
C. Persona handoff (autoplay reaches a beat where the next actor differs) Autoplay pauses · handoff cue appears · avatar gets purple glow ring Lead-mode does NOT re-enter (gate is mid-beat). Auto-collapse pauses while gate is up.
D. User dismisses lead-mode (click on stage / arrow / space / 3–6.5s timeout) Lead-mode exits · sidebar shrinks back to rail · headline drops to Inter 22px · lead hint fades Persona card stays. Step label stays. Headline + lede text stay (smaller).

4 · Vignette mapping — which event each block belongs to

This is the table to fill in if any of these are wrong. Each block has an intended trigger; if the screenshot you flagged shows a block animating against the wrong trigger, that's the row to mark.

Vignette A · Scene change

Tab click or end-of-scene autoplay

Should animate: 3 (headline swells), panel pop, 5 armed

Should bounce: 1 only if persona identity differs across the seam

Should NOT appear: 6 (the gate is for mid-scene handoffs, not scene boundaries)

Vignette B · Beat advance

Next, Prev, →, ←, or autoplay tick within a scene

Should animate: 3 (text swap), 4 (text swap), panel re-pop

Should bounce: 1 only if persona identity differs from previous beat

Should NOT appear: 6 unless the next beat is a different actor

Vignette C · Persona handoff

Autoplay reaches a beat where the actor changes

Should appear: 6 (handoff cue) · avatar glow ring on 1

Should pause: autoplay tick · auto-collapse timer

Should NOT animate: panel re-pop (the gate is what matters; the panel was already there)

Vignette D · Lead-mode dismiss

Click stage, arrow keys, space, or 3–6.5s timeout

Should animate: panel collapse, headline shrink, 5 fade out

Should NOT animate: 1 (persona card stays put), 6 (gate is unrelated)

Vignette E · Cold boot

First load of story-shell.html

Should appear: sidebar in narrow rail state with persona card, step label, headline, lede

Should NOT animate: panel pop, 1 bounce, 6 cue (post-fix on 2026-04-30 — passes silent:true)

Vignette F · Async usecase load

?usecase=… bundle resolves ~50ms after page open

Should appear: the new scene's headline + lede swap in silently

Should NOT animate: panel pop on top of the user (post-fix on 2026-04-30 — passes silent:true)

5 · Locked-in decisions 2026-04-30

Five behavior questions resolved. Each row is the rule going forward.

ID Question Locked-in answer
D1 When does the handoff cue appear? Only after the auto-collapse window elapses. Autoplay pauses immediately at the handoff beat, but the lead-mode panel reads on its own first. The cue lands when the read window (3–6.5s, sized to the beat's word count) would have collapsed the panel — replacing the collapse with a call-to-action.
D2 Lead hint in non-lead-mode states? No. The "CLICK ANYWHERE TO BEGIN" hint stays scoped to lead-mode. No idle-timer hint elsewhere on the page.
D3 Avatar glow on every persona change? Yes. The chip's avatar pulses purple for ~0.95s whenever persona identity changes — manual Next/Prev advances included, not only the autoplay-pause gate. The sustained .is-handoff-pending glow is reserved for the gate; the new .id-pulse is the short signal for "actor just changed."
D4 Italic headline for system-side beats? No. When persona[data-side="system"], the lead-mode headline drops to non-italic Inter at 34px. Editorial Fraunces italic stays reserved for human actors. The shell mirrors persona side onto itself via data-persona-side for CSS to hook.
D5 Side-badge ("Advisor"/"Client") on named personas? No. Badge stays anonymous-only. Named personas rely on the role line to communicate side.
