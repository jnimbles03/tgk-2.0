Bespoke mode, made safe

Out of the Box now protects AEs with a three-question guided path. This brief covers the second half of the ask: how to make the iMovie-style Bespoke builder easy enough that an AE can assemble and pitch a custom demo without getting into trouble.

2026-06-11 · Prepared for Jimmy · Companion to the Out of the Box mode shipped in builder.html

The problem, precisely

A 30–45 year-old SaaS AE can absolutely narrate a story over a staged, animated demo. What they cannot reliably do is compose one from raw parts: pick 5 coherent scenes from a pool of 13+, order them into an arc, name the tenant, pick a brand color, and know what to say on every beat. The risk isn't capability — it's that free composition has too many ways to produce something incoherent, off-brand, or story-broken in front of a customer.

The market has converged on a consistent answer, and it's the same answer iMovie Trailers shipped in 2010: don't reduce the power, reduce the degrees of freedom.

What the market does

Six patterns recur across the demo-platform category (Storylane, Walnut, Reprise, Consensus, Demostack, Navattic, Supademo) and adjacent guided-creation tools:

PatternWho does itThe move
Template-first, never blankReprise, Guideflow, iMovieCreation always starts from a vetted template; "blank canvas" isn't an entry point. Reps personalize, they don't originate.
Fixed storyboard, swappable slotsiMovie TrailersThe arc is locked (the trailer's beats); the user only fills placeholders that say what belongs there ("action shot," "closeup"). Duration, transitions, and music are automatic.
Token personalizationWalnut, Demostack, TrupeerLogo, prospect name, colors, and data are variables swapped across the whole demo in one action — a brand kit, not per-screen edits.
Vetted library + governanceConsensus, Guideflow, DemostackReps assemble from a curated library; approval workflows gate what becomes customer-facing. Without governance, "demo libraries drift" — one-off versions, stale messaging.
Presenter mode with talk trackStorylane, DemostackLive-demo view pairs each step with talking points and navigation guardrails so the rep "stays in control without going off-script."
AI assembly within boundariesWalnut, Demostack, SupademoAI drafts the personalized demo (by industry, persona, use case) but only inside approved messaging and visual standards; the rep reviews a draft, not a blank page.
The unfair advantage

Every one of these patterns has a half-built counterpart already in this repo. The recommendations below are mostly wiring, not new construction — the lego thesis (blocks × verticals, content/theme split) is exactly the architecture these products charge per-seat for.

Recommendations, ranked

R1Talk-track presenter viewLow effortHighest impact

A presenter strip (or second-window view) that shows the current beat's narration lede plus a one-line "why this matters" as the AE scrubs. The AE's job collapses from "compose and narrate" to "advance and read ahead." This is the single biggest de-risking move because it fixes the live-pitch moment, which is where trouble actually surfaces.

Reuses: authored beat ledes in the shell's VERTICALS map · demo-stage virtual clock + beat-snap keyboard scrubbing. Net-new: one overlay panel.

R2Template-first Bespoke (kill the blank canvas)Low effortHigh impact

Entering Bespoke should open a "start from…" sheet: the same 3-question OOTB picker, but landing in the editable timeline instead of the launch card. Free-build from empty remains available behind a "start empty" link for Steven/Jesse-class users. Reps personalize a working demo; they never originate from zero.

Reuses: the OOTB map just shipped · existing audit-mode hydration (?vertical=&usecase=). Net-new: one handoff button ("Customize in Bespoke") on the OOTB launch card.

R3Storyboard slots instead of free timelineMedium effortHigh impact

The iMovie move, and the UX realization of the lego thesis. For AEs, replace drag-anything-anywhere with five fixed slots labeled by narrative function (Hook · Identity at the door · The work · The signature moment · The workspace). Each slot offers only the catalog blocks valid for that slot × the chosen vertical. Order is not editable; content within a slot is. Power users keep the free timeline behind the same "advanced" flag as R2.

Reuses: 5-scene spine · USECASE_SCENE_KEYS slot semantics · demo-catalog.js (needs a slot + vertical tag per block — the per-vertical-block-files → one-block-plus-skin gap is the same work).

R4Timeline lint (guardrails as validation)Low effortMedium impact

Whatever the AE assembles, lint it before Generate URL: must end in an executed agreement or Workspace moment; an identity-verification scene can't dangle without a downstream action; warn when scene count exceeds the spine. Friendly one-line fixes, not blocks. This also closes a known thesis gap: a lint rule "fraud-fabric story should surface the Certificate of Completion" forces the CoC artifact into view.

Reuses: timeline state in the builder store. Net-new: ~a dozen rules and a message strip.

R5Brand packs + one-touch token sweepLow–medium effortMedium impact

Stage 2 already has the right fields; the friction is filling them credibly. Ship named packs per sub-vertical (tenant, palette, SoR, customer line — e.g. "Cedar Federal · nCino" or "Sentinel · Guidewire") so branding is one click, and apply the swap across every scene at once. Optional later: paste a customer URL and prefill name + palette server-side.

Reuses: stage-2 fields + scaffold defaults + SoR datalist · SoR badge treatment (Guidewire/nCino) · theme.json split. Net-new: a packs array.

R6Vetted team library with a promotion loopMedium effortMedium–high impact

Saved bespoke builds (already persisted via the builder store / /demos API) get a team library view: AEs browse and remix each other's demos rather than raw blocks. A "vetted" badge is granted through the existing audit queue — the same submit → review → apply loop audit.html already runs for content edits. This is the governance answer to demo drift, and it compounds: every good bespoke demo becomes tomorrow's template.

Reuses: /demos API (GET/POST live; PATCH/DELETE planned) · audit queue + /admin/audit review surface · custom-player share links.

R7"Describe the customer" AI draftHigher effortHigh ceiling

A textarea in Bespoke: paste discovery notes or a customer one-liner; classify → match → assemble drafts a timeline from catalog blocks, pre-branded via R5, landing in the slot view (R3) for review. This is Walnut's "AI personalization within approved boundaries" — and the boundary system (canonical blocks, lint, vetted library) is exactly what R1–R6 build. Sequence it last so the AI composes inside guardrails rather than around them.

Reuses: studio pipeline stages + verbatim prompts (builder/prompts/v1/) · Gemini wiring + fake mode · /demos API for persistence. Net-new: one prompt + one endpoint + glue.

Suggested sequence

  1. Now (days): R2 template-first handoff button, R1 presenter talk-track strip, R4 lint rules. Together these make today's Bespoke pitch-safe without touching the timeline mechanics.
  2. Next (1–2 weeks): R5 brand packs, then R3 storyboard slots — R3 is the structural change, and it's also the forcing function to finish the block-tagging work the lego thesis already calls for.
  3. Later: R6 library + promotion loop, then R7 AI drafting on top of everything.

A useful test for every change: an AE should never face a decision the catalog could have made for them, and never present a screen nobody vetted. OOTB satisfies that today; R1–R7 extend the same property into Bespoke.
